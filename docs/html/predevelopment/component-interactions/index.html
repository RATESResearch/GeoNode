Determination of Component Interactions¶

Three components make RGVFlood application ecosystem specific to the LRGV (Figure: 10). Hydrologic data specific to the LRGV, includes national & state level data, along with locally collected data as well as local forcings from RTHS.us. This RGV Hydrography is stored in a REON PostgreSQL database (REON/db). The RGV Water Model (REON/WM), is driven by RGV Hydrography and tuned using local forcings from RTHS.us. The interface to REON.cc, tuned to the specific needs of the LRGV users, is accessed from RGVFlood.com.

RGVFlood relies on the REON cyberinfrastructure, which includes the RTHS.us network serving forcing data, station metadata and flood early warning information from the RTHS stations. REON/WM refers to the ecosystem of hydrologic, hydraulic & stormwater forecast models. These models are defined in Tiers - with Tier I representing the continuous running real-time regional hydrologic forecast model (WRF-Hydro), Tier II being an on-demand hydraulic model (HEC-RAS), applied to a subset of the region, Tier III being an on-demand hydrologic charactgerization model (HEC-HMS), and Tier IV being urban stormwater models such as SWMM. Primary user interaction with RGVFlood is through the REON.cc cybercollaboratory, whic includes a framework of analytic & decision support applications, pulling data through the GeoNode Django interface as needed. The entire REON cyberinfrastructure is data driven, relying on a PostgreSQL database server (REON/db) with PostGIS extensions, storing REON specific data for RTHS, REON/WM and REON.cc.

Geospatial content and collaborative services are provided by a GeoNode instance. GeoNode relies on a geospatial data server (GeoServer) to host the data layers, and a Python web framework (Django) to serve the base user interface. The Django user interface is extended with the REON.cc web applications. As with REON, the GeoNode ecosystem is served by a PostgreSQL database server (GeoNode/db) with PostGIS extensions, storing GeoNode Django and GeoServer data. Django uses a task scheduling and messaging application (Celery) to maximize parallel task processing. The RGVFlood instance of GeoNode relies on the NGINX high performance web server to serve its components.
